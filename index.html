JSON Prompt Engineering

A Playbook for VS Code + GitHub Copilot

Scope: VS Code + GitHub Copilot (Ask, Edit, Agent). Use this as a reusable playbook for structured, repeatable prompts.
Created by - Anubhav Chaudhary

1) What is JSON prompting?

JSON prompting expresses your request as a structured JSON object instead of free text. It’s machine-readable, repeatable, and great for PRs/wikis. Use it for complex, multi-step tasks, team-shared prompts, and Agent guardrails.

Attach precise context via Copilot Chat symbols: #selection, #file, #codebase, @workspace.

2) Canonical JSON prompt (template)

Copied!
{ "role": "senior-java-reviewer", "goal": "Refactor and document selected code without changing behavior", "context": { "language": "java", "framework": "spring-boot", "versions": { "java": "17", "springBoot": "3.3" } }, "input": { "code_ref": "#selection", "file": "#file" }, "constraints": [ "no API signature changes", "add null-safety and logging", "preserve time/space complexity" ], "steps": [ "analyze", "propose_changes", "apply_changes" ], "output_format": { "type": "patch", "include_rationale": true }, "acceptance_criteria": [ "build passes", "existing tests pass", "no deprecated Spring APIs" ], "risk_checklist": [ "thread-safety", "transactions", "error-mapping" ] }

How to send in VS Code:

  • • Ask mode: paste JSON after your /explain prompt (use @workspace + #selection).
  • • Edit mode: select code, open Edit, paste JSON as the instruction.
  • • Agent mode: paste JSON; add scope/acceptance tests; review diffs.

Tip: JSON has no comments—use a "_note" field for annotations.

A) Generate JUnit tests for a service (Ask → /tests)

E) Agent: Add CORS + regression tests (multi-file)

Copied!
{ "role": "feature-agent", "goal": "Enable CORS for http://localhost:5173 across REST endpoints and add tests", "scope": { "allow": [ "src/main/java/**", "src/test/java/**" ], "deny": [ "pom.xml" ] }, "constraints": [ "Spring Boot 3 idioms only" ], "output_format": { "type": "multi-patch" }, "acceptance_criteria": [ "unit tests assert CORS headers on /api/*", "`mvn -q -DskipTests=false test` passes", "README section added" ] }

4) Patterns & tips for robust JSON prompts

  • • Keep keys stable & meaningful: goal, input, constraints, output_format, acceptance_criteria, scope, examples, steps.
  • • Pin context with #selection, #file, @workspace rather than pasting huge code blocks.
  • • Use enums for outputs: output_format.type="code"|"patch"|"summary"|"markdown"| "json" | "test-file".
  • • Add guardrails: scope.allow/deny, must_not, security (e.g., "must_not": ["alter SQL schema"]).
  • • Few-shot inside JSON: add 1–3 examples to define the pattern.
  • • No comments in JSON; use "_note" for annotations. Avoid trailing commas.
  • • Limit size: prefer references + acceptance tests over dumping entire files.

5) Mini schema (informal)

Copied!
{ "role": "string", "goal": "string", "context": { "language": "string", "framework": "string", "versions": { "java": "string" } }, "input": { "code_ref": "#selection|#file|...", "file": "string", "terminal": "#terminalLastCommand" }, "constraints": [ "string" ], "steps": [ "string" ], "examples": [ { "name": "string", "given": {}, "expect": "string" } ], "output_format": { "type": "code|patch|summary|markdown|json|test-file", "include_rationale": true }, "scope": { "allow": [ "glob" ], "deny": [ "glob" ] }, "acceptance_criteria": [ "string" ], "risk_checklist": [ "string" ], "_note": "optional" }

6) Anti-patterns to avoid

  • • Mixing free-form paragraphs with JSON—stick to pure JSON objects.
  • • Contradictory constraints (e.g., "no API changes" + "add parameter").
  • • Missing acceptance criteria (no way to verify success).
  • • Pasting huge code blobs (use #selection/#file instead).

7) QA checklist before sending

  • • JSON is valid (no comments/trailing commas)
  • • Scope and constraints are clear
  • • Output type is defined
  • • Acceptance criteria are testable
  • • Proper #selection/#file/@workspace references added
